Practice Case Study · ITSM Process Design

Incident & change management: closing the loop between firefighting and change control

A self-directed exercise applying ITIL incident and change management practice (the same discipline behind my ServiceNow Business Process Consultant certification) to a common IT service management failure mode: incidents get fought and fixed, but the changes that caused them are never reviewed or tracked, so the same outages repeat. Data is synthetic, built to reflect realistic ITSM benchmark patterns.

Scenario · Mid-size company IT service desk Method · Swimlane mapping, gap analysis, As-Is/To-Be redesign Tools · ITIL incident & change management, ServiceNow process design, KPI modeling
Problem

Incidents get fought, but the changes behind them never get reviewed

A mid-size company's IT incidents are reported through whatever channel is fastest (Slack, email, a phone call to whoever answers), with no ticket and no severity classification, so a login glitch and a full outage queue the same way. Under time pressure, technical fixes are pushed straight to production without going through the Change Advisory Board, because the only review path that exists is too slow for anything urgent. Nobody formally reviews incidents after the fact, so the same change-related failures repeat without anyone tracing them back to their root cause.

Gap Analysis

What's actually causing the repeat outages

Process gap

Incidents arrive through multiple informal channels with no single system of record, so there's no reliable SLA clock or incident history to analyze.

Fix: one ticketed intake channel, timestamped automatically at report
Governance gap

Triage doesn't classify severity, so a critical outage can sit in the same queue as a password reset request.

Fix: Sev 1–4 framework tied to business impact, with automatic escalation for Sev 1
Governance gap

The only change review path (standard CAB) is too slow for urgent fixes, so teams skip review entirely rather than use an accelerated path. The gap isn't that urgent changes get reviewed fast; it's that they get reviewed at all.

Fix: formal Emergency Change Advisory Board (ECAB), expedited rather than skipped
Process gap

There's no post-incident review step, so root cause is never traced back to the change that caused it, and the same failure mode repeats undetected.

Fix: mandatory post-incident review within 48 hours for Sev 1/Sev 2, logged against the change record
Implementation Approach

Governance fix, phased by cost, not a forced platform purchase

The instinct is often to recommend a full ITSM platform first. That's usually the wrong starting point. Most of this gap is a governance and discipline problem, not a tooling problem, and it can be closed before any new software is purchased.

Tier 1 · $0

Process discipline

Define a severity framework and a single intake channel using whatever's already in place; a shared inbox with a mandatory template works as a starting point. No new software required.

Tier 2 · Low cost

Lightweight ECAB cadence

Stand up a recurring 15-minute virtual ECAB check-in for emergency changes using the calendar and chat tools already in use, a fast, mandatory checkpoint rather than a new platform.

Tier 3 · Capital investment

ITSM platform (e.g. ServiceNow)

Justified once volume and complexity outgrow manual tracking: automated ticketing, workflow routing, and a searchable known-error database. A later phase, not the starting recommendation.
